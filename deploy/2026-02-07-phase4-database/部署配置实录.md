### 2.2 部署到 home-node

#### 第一次部署（失败）
```bash
ssh home-node "cd /srv/soulpal-service/deploy && bash deploy.sh"
```

**失败原因**:
应用尝试连接 `localhost:5432` 而非 Docker 网络中的 `postgres:5432`

**错误日志**:
```
Caused by: org.postgresql.util.PSQLException: Connection to localhost:5432 refused
Caused by: java.net.ConnectException: Connection refused
```

#### 问题诊断
检查 `application-dev.yml` 发现硬编码：
```yaml
datasource:
  url: jdbc:postgresql://localhost:5432/soulpal
```

**根本原因**:
docker-compose.yml 设置了 `DB_HOST=postgres`，但配置文件未使用环境变量。

#### 修复方案
修改 `application-dev.yml`:
```yaml
datasource:
  url: jdbc:postgresql://${DB_HOST:localhost}:${DB_PORT:5432}/${DB_NAME:soulpal}
  username: ${DB_USER:soulpal}
  password: ${DB_PASSWORD:soulpal_dev_2026}
```

**提交修复**:
```bash
git add soulpal-api/src/main/resources/application-dev.yml
git commit -m "fix: 修复 application-dev.yml 数据库连接配置"
git push origin main
```

#### 第二次部署（成功）

## 四、技术要点记录

### 4.1 环境变量配置原则
**问题**: application-dev.yml 硬编码 localhost 导致 Docker 环境无法使用
**解决**: 所有配置文件（包括 dev）都应使用 `${VAR:default}` 占位符
**规范**:
```yaml
url: jdbc:postgresql://${DB_HOST:localhost}:${DB_PORT:5432}/${DB_NAME:soulpal}
```

### 4.2 Docker 网络通信
- 容器间通信使用**服务名**（`postgres`）而非 `localhost`
- docker-compose.yml 通过环境变量注入服务名：
  ```yaml
  environment:
    DB_HOST: postgres  # 指向 postgres 服务
  ```

### 4.3 DDD Light 架构验证
✅ **Domain 层**纯净（无框架注解）
✅ **Infrastructure 层**隔离（SessionDO 包含 MyBatis-Plus 注解）
✅ **Converter 模式**正确实现（LocalDateTime ↔ Instant + UTC 时区）
✅ **依赖倒置**原则遵守（Application 依赖 Repository 接口）

### 4.4 MyBatis-Plus 关键配置
```yaml
mybatis-plus:
  global-config:
    db-config:
      id-type: ASSIGN_UUID  # 分布式友好的 UUID 生成
      logic-delete-field: deleted
      logic-delete-value: true
      logic-not-delete-value: false
  configuration:
    map-underscore-to-camel-case: true
    log-impl: org.apache.ibatis.logging.slf4j.Slf4jImpl
```

## 九、经验教训

1. **环境变量配置**: 即使是开发环境配置，也应使用 `${VAR:default}` 占位符，以支持 Docker 部署
2. **Docker 网络**: 容器间通信必须使用服务名，而非 localhost
3. **Flyway 版本管理**: 让 Spring Boot BOM 管理依赖版本，避免版本号不存在的问题
4. **时区处理**: 统一使用 `ZoneId.of("UTC")`，避免 systemDefault() 带来的不确定性
5. **DDD Light 分层**: 严格遵守分层原则，DO 和 Entity 分离，确保 Domain 层纯净
