# ADR-0001 技术选型定稿 v2.0

- **状态**：Accepted（替代 v1.0）
- **版本**：2.0
- **适用范围**：宠物项圈 App 后台工程（MVP → 可扩展长期演进）
- **核心目标**：轻量起步、快速迭代、团队可维护、模块化单体起步、具备微服务演进路径、支持实时能力（WebSocket）、API 契约（OpenAPI）

---

## 变更说明（v1.0 → v2.0）

| 变更项 | v1.0 | v2.0 | 变更理由 |
|--------|------|------|----------|
| 语言 | TypeScript | **Java 21 LTS** | 公司技术栈统一、人才储备充足 |
| 框架 | NestJS | **Spring Boot 3.x** | 公司成熟使用、团队可维护 |
| ORM | Prisma（候选） | **MyBatis-Plus** | 团队熟悉、SQL 可控 |
| 构建工具 | npm | **Maven** | 公司标准 |

> 决策背景：项目后续将交接给公司 Java 团队维护，降低交接成本和长期维护风险。

## 1. 技术选型结论

### 1.1 语言与框架
- **语言**：Java 21 LTS
- **后端框架**：Spring Boot 3.x
- **构建工具**：Maven（多模块项目）
- **架构策略**：模块化单体（Modular Monolith）起步，按业务边界模块化设计，支持未来拆分为微服务

### 1.2 通信协议与接口契约
- **对外接口**：HTTP REST
- **接口契约**：OpenAPI 3.0（使用 springdoc-openapi，路径版本化 `/api/v1`）
- **实时通信**：WebSocket（Spring WebSocket + STOMP 或原生 WebSocket）
- **说明**：
  - OpenAPI 仅覆盖 HTTP REST 接口
  - WebSocket 事件协议单独文档化

### 1.3 数据与中间件
- **业务数据库**：PostgreSQL
- **缓存/分布式能力**：Redis
  - 缓存
  - 限流（Spring Cloud Gateway / 自定义）
  - 会话/状态管理
  - WebSocket 多实例扩展预留
  - 队列能力预留（后续可引入 RabbitMQ / Kafka）
- **数据访问层**：MyBatis-Plus
- **数据库迁移**：Flyway

### 1.4 异步任务与后台处理
- **进程形态**：API 服务 + Worker 服务（逻辑隔离）
- **职责**：
  - API：HTTP/WS 请求处理、业务编排入口
  - Worker：异步任务、数据消费、派生计算、通知处理
- **异步任务框架**：
  - 简单任务：Spring @Async + ThreadPoolTaskExecutor
  - 定时任务：Spring @Scheduled
  - 复杂任务：预留 XXL-JOB 或 Spring Cloud Task 扩展
- **一致性策略**：
  - v1.0 以数据库事务为主（@Transactional）
  - 预留 Outbox / 事件驱动扩展边界（v1.0 不强制实现）

### 1.5 设备与物联网平台对接（未来规划）
- **设备连接与托管**：长期规划托管至第三方物联网云平台（如 AWS IoT）
- **时序数据**：
  - 由 IoT 平台或数据管道直接入库（平台侧负责 ingest）
  - 本工程侧负责数据消费、业务聚合与推送
- **边界说明**：
  - 本工程不承担大规模设备长连接 Broker 职责
  - 聚焦"用数 / 取数 / 业务编排 / 实时推送（WS）/ 管理接口（HTTP）"

### 1.6 部署形态（与技术选型相关）
- **起步部署**：家庭节点（Windows + WSL2）+ Cloudflare Tunnel + Docker Compose
- **运维入口**：SSH（`ssh home-node`）
- **演进路径**：
  - 单机容器（当前）
  - 多实例水平扩展
  - 迁移到 VPS / 云（Cloudflare 入口不变，仅替换 origin）
  - 托管容器 / Kubernetes（可选）
- **原则**：
  - 所有核心组件可容器化
  - 配置通过环境变量管理（Spring Boot 外部化配置）
  - 架构不绑定特定云厂商，支持平滑迁移

## 2. 选型理由

1. **团队可维护**
   Java + Spring Boot 是公司成熟使用的技术栈，团队熟悉度高，代码审查、问题排查、知识传承成本低。

2. **人才储备充足**
   公司现有人才匹配均为 Java，后续招聘和扩展团队无障碍。

3. **轻量起步但边界清晰**
   模块化单体在早期降低复杂度，同时为未来拆分微服务预留清晰边界。

4. **通信模型成熟稳健**
   HTTP + OpenAPI 作为主契约，WebSocket 作为实时补充，符合主流 App 后台模式。

5. **顺应 IoT 托管趋势**
   将设备连接与时序数据采集交给第三方平台，本工程聚焦业务价值层。

6. **AI 辅助开发支持**
   Claude 对 Java / Spring Boot 的理解成熟，代码生成质量不亚于 TypeScript。

## 3. 架构原则（强约束）

### 3.1 分层原则
- **Interfaces / API**：@RestController、DTO、Request/Response、协议适配（HTTP / WebSocket）
- **Application**：@Service（UseCase）、@Transactional（事务边界）、业务编排
- **Domain**：Entity、Value Object、Domain Service（纯 POJO，无 Spring / JPA 注解）
- **Infrastructure**：@Repository 实现、MyBatis Mapper、Redis Client、外部平台适配

**禁止**：
- Controller 直接实现核心业务逻辑
- Domain 层直接依赖 Spring、MyBatis、Redis、第三方 SDK
- 跨模块直接访问对方的 Domain 层

### 3.2 契约优先
- OpenAPI 文件必须纳入版本控制
- 接口变更必须同步更新 OpenAPI（springdoc-openapi 可自动生成）

### 3.3 可演进边界
- 模块之间通过 Application 层接口交互
- API 与 Worker 进程边界清晰，可独立扩展与拆分
- 跨模块引用使用 ID / Value Object，不直接传递实体

## 4. 风险与缓解措施

### 4.1 依赖管理风险
- **风险**：Maven 依赖冲突、版本兼容性
- **缓解**：
  - 使用 Spring Boot BOM 统一版本管理
  - 父 POM 锁定核心依赖版本
  - CI 中引入依赖与安全扫描（OWASP Dependency-Check）

### 4.2 WebSocket 扩展风险
- **风险**：多实例场景下连接路由与广播复杂
- **缓解**：
  - v1.0 先支持单实例
  - 预留 Redis Pub/Sub / Spring Session 扩展方案

### 4.3 一致性与复杂业务演进风险
- **风险**：后期引入事件驱动可能需要重构
- **缓解**：
  - Application 层预留事件发布/Outbox 扩展点
  - 关键写操作预留幂等规范

### 4.4 MyBatis 与 DDD 融合风险
- **风险**：MyBatis 偏数据驱动，与纯 DDD 有一定摩擦
- **缓解**：
  - Domain 层保持纯 POJO
  - Infrastructure 层负责 Domain ↔ DO（Data Object）转换
  - 不在 Domain 实体上加 MyBatis 注解

---

## 5. 非目标（v1.0 明确不做）

- 不自研 MQTT / 设备长连接 Broker
- 不在 v1.0 引入完整微服务治理体系（注册中心、服务网格、全量 MQ）
- 不强制引入 AsyncAPI（WebSocket 事件仅要求文档化）
- 不引入 Spring Cloud 全家桶（v1.0 保持 Spring Boot 单体）
