# review-checklist 生成指南

本文档指导 AI 如何从多个 Tech Pack 汇总生成联合快审要点。

---

## 文件结构

review-checklist 固定 6 个章节：

```
§1 文档清单
§2 待确认决策项
§3 跨批次联动点
§4 规则覆盖完整性检查
§5 审查检查清单
§6 快审结论
```

---

## §1 文档清单

**数据来源**：所有批次的 Tech Pack

汇总表格：

| 列 | 来源 |
|----|------|
| 批次 | P0 / P1 / P2 |
| 文档路径 | Tech Pack 文件名 |
| API 数量 | 从 §4.1 接口清单统计，标注 ID 范围 |
| 新建表 | 从 §3.1 核心实体提取 |
| 测试用例 | 从 §5.1 统计，标注 TC-ID 范围 |
| 决策项 | 从 §3.2 决策项表统计，标注 D-xx 范围 |

---

## §2 待确认决策项

**数据来源**：所有批次 Tech Pack §3.2 和其他章节中的决策项

### 分组方式

按批次分组，每组一个表格：

```markdown
### P0 决策（D-01 ~ D-xx）

| # | 决策项 | 建议 | 确认结果 |
|---|--------|------|----------|
| D-01 | <决策问题> | <AI 建议 + 简要理由> | |
```

**关键要求：**

- 「确认结果」列留空，由人评审时填写
- 建议列要包含足够的上下文让人做判断，但不要过长
- 如果某个决策有明显的最佳方案，在建议中说明理由
- 如果存在多种可行方案，简要列出各方案的优劣

---

## §3 跨批次联动点

**数据来源**：所有批次 Tech Pack §8（变更联动）

汇总表格：

| 列 | 说明 |
|----|------|
| 联动ID | CL-xx |
| 方向 | 如 `P2 → P0`，表示 P2 会改造 P0 的内容 |
| 内容 | 具体的联动内容 |
| 影响范围 | 受影响的代码/接口/表 |

在表格下方追加一行**审查重点**，提示评审者关注的核心问题。

---

## §4 规则覆盖完整性检查

**数据来源**：PRD §5（BR-xx）、§6（AC-xx）+ 所有 Tech Pack §2（规则映射）+ §5（测试清单）

### 4.1 PRD 业务规则 → Tech Pack 映射

逐条检查每个 BR-xx 是否在某个批次的 Tech Pack §2 中有映射：

| 列 | 说明 |
|----|------|
| 规则ID | BR-xx |
| 规则摘要 | 从 PRD §5 提取 |
| 批次 | 该规则属于哪个批次 |
| Tech Pack 是否覆盖 | ✅ + TC-ID / ❌ 未覆盖 |

**如果有 ❌ 未覆盖的规则，标注为 ERROR，生成结束后在输出摘要中警告。**

### 4.2 PRD 验收项 → 测试用例映射

逐条检查每个 AC-xx 是否有对应的 TC：

| 列 | 说明 |
|----|------|
| AC-ID | AC-xx |
| 验收描述 | 从 PRD §6 提取 |
| 批次 | 该 AC 属于哪个批次 |
| 对应测试 | TC-ID / 集成测试覆盖 |

**注意**：有些 AC 可能没有独立的 TC，而是被集成测试覆盖。这种情况标注「集成测试覆盖」即可，不算 ERROR。

---

## §5 审查检查清单

**数据来源**：AI 基于所有 Tech Pack 内容综合分析生成

按维度分组，每组包含可勾选的检查项：

### 5.1 数据库设计

重点检查：
- 表结构完整性（字段是否覆盖所有规则）
- 软删除 / 硬删除策略是否合理
- 索引设计是否满足查询场景和体量假设
- 跨批次加字段的兼容性

### 5.2 API 设计

重点检查：
- 路径命名一致性（RESTful 风格）
- 鉴权要求正确性
- 错误码覆盖完整性
- 关键接口的校验顺序是否与 PRD 一致

### 5.3 跨批次一致性

重点检查：
- 前置批次是否足够预留扩展点供后续批次改造
- 后续批次改造前置代码的方案是否可行
- Flyway 迁移脚本是否能兼容已有数据

### 5.4 测试覆盖

重点检查：
- BR 规则全覆盖
- AC 验收标准全覆盖
- 边界场景是否充分

**格式**：每个检查项前使用 `- [ ]`（未检查的 checkbox），评审时由人勾选。

---

## §6 快审结论

**生成时留空**，定稿时（模式 B）填写：

```markdown
## 6. 快审结论

| 日期 | 结论（通过/退回） | 关键问题 | 下一步 |
|------|--------------------|----------|--------|
| | | | |

**通过后下一步**：进入阶段 D — AI 主导开发，按 P0 → P1 → P2 顺序开发。
```

---

## 质量自检

生成 review-checklist 后，AI 自动执行以下自检：

| 检查项 | 规则 |
|--------|------|
| 决策项完整 | 所有批次的 D-xx 都在 §2 中列出 |
| 联动点完整 | 所有批次的 CL-xx 都在 §3 中列出 |
| BR 全覆盖 | §4.1 中没有 ❌ 项 |
| AC 全覆盖 | §4.2 中所有 AC 都有对应测试或覆盖说明 |
| checkbox 未勾选 | §5 所有检查项都是 `- [ ]` 状态（留给人审） |
| §6 为空 | 快审结论留空 |
