# Tech Pack 各章节生成指南

本文档指导 AI 如何从 PRD 和代码上下文生成 Tech Pack 各章节内容。

---

## §1 输入基线

**数据来源**：PRD §5（业务规则）、§6（验收标准）、§7（交付约束与分期策略）

**生成规则：**

1. **PRD 版本**：从 PRD 文档头提取版本号
2. **规则 ID 范围**：从 PRD §7.2 对应批次的「必须映射的规则」提取 BR-xx 列表
3. **验收项范围**：从 PRD §6 中筛选属于当前批次的 AC-xx
4. **前置条件**：
   - P0：项目基础设施（根据 MEMORY.md 中的 Progress 判断）
   - P1+：前置批次的 Tech Pack 已基线
5. **功能范围**：从 PRD §7.2 对应批次的「主要交付物」提取
6. **非目标**：从 PRD §7.2 中排除的功能 + 其他批次的功能
7. **体量假设**：从 PRD 的约束中提取或根据项目背景推导

---

## §2 规则映射总表

**数据来源**：PRD §5（业务规则）

**生成规则：**

对当前批次涉及的每条 BR-xx，逐条分析并填写：

| 列 | 生成方法 |
|----|----------|
| PRD规则ID | 直接取 BR-xx |
| DB约束 | 分析规则的数据层面约束：唯一索引、CHECK 约束、外键、软删除字段等 |
| API行为 | 分析规则的接口行为：校验逻辑、业务流程、返回值处理 |
| 测试点ID | 分配 TC-ID（从前置批次最大 TC-ID 之后开始） |
| 状态 | 统一填「待开发」 |

**关键原则：**

- 一条 BR 可能映射多个 TC（不同场景），用逗号分隔
- DB 约束要具体到字段名和约束类型
- API 行为只写关键逻辑，不写实现细节
- 如果某条规则没有 DB 约束（纯应用层逻辑），标注「无 DB 约束」

## §3 数据库设计

**数据来源**：§2 规则映射 + PRD §3（核心实体与关系）

### §3.1 核心实体

从 §2 中提取需要新建或修改的表：

| 列 | 生成方法 |
|----|----------|
| 表名 | 使用 snake_case 复数形式（如 `users`、`pets`） |
| 用途 | 一句话说明 |
| 备注 | 新建 / 修改（说明改什么） |

### §3.2 关键约束与决策

**只写需要人确认的约束**，按表分组列出：

- 唯一性约束（哪些字段需要唯一索引）
- 软删除策略（是否需要、保留周期）
- 事务边界（哪些操作必须在同一事务）
- 级联删除策略（物理删除还是级联软删）

**决策项（D-xx）生成规则：**

遇到以下情况时生成决策项：
1. 有多种可行的技术方案，各有利弊
2. 涉及第三方依赖选型
3. 对后续批次有预留影响的设计
4. 性能 vs 简洁性的权衡

每个 D-xx 必须包含：
- 决策问题（清晰的问句）
- 建议方案 + 理由
- 状态：「待确认」

## §4 API 设计

**数据来源**：§2 规则映射 + §3 数据库设计

### §4.1 接口清单

| 列 | 生成方法 |
|----|----------|
| API-ID | 从前置批次最大 API-ID 之后开始 |
| 方法 | RESTful 语义：POST（创建）、GET（查询）、PUT（全量更新）、PATCH（部分更新）、DELETE |
| 路径 | `/api/v1/<resource>`，使用 kebab-case 复数 |
| 用途 | 一句话说明 |
| 鉴权 | 是/否（注册和登录不需要，其他默认需要） |
| 优先级 | 当前批次标识 |

**路径设计规范：**

- 资源嵌套最多两层：`/api/v1/teams/{teamId}/members`
- 非 CRUD 操作用动词子路径：`/api/v1/pets/{id}/restore`
- 查询参数用于过滤和分页，不出现在路径中

### §4.2 关键约束

只写 AI 开发时无法自行判断的约束：
- 多步校验的顺序
- 并发冲突处理策略
- 幂等性要求
- 频率限制

### §4.3 鉴权方案

**仅首批次（P0）需要详细写**，后续批次标注「复用 P0 鉴权方案」。

读取现有代码的认证模块结构（AuthInterceptor / TokenProvider），生成：
- 认证方式
- 认证流程
- 排除路径列表（不需要鉴权的 API）

### §4.4 错误码规范

**命名规则**：`<MODULE>_<CATEGORY>_<SEQ>`

| 分类 | 示例 |
|------|------|
| AUTH | `USER_AUTH_001`（认证失败） |
| VALIDATION | `USER_VALIDATION_001`（参数校验失败） |
| BUSINESS | `USER_BUSINESS_001`（业务规则违反） |
| NOT_FOUND | `USER_NOT_FOUND_001` |

确保：
- 同一语义不重复定义
- HTTP 状态码与语义匹配（400 参数错误、401 未认证、403 无权限、404 不存在、409 冲突）

## §5 测试清单

**数据来源**：§2 规则映射 + §4 API 设计

### §5.1 核心测试点

从 §2 的测试点 ID 展开：

| 列 | 生成方法 |
|----|----------|
| TC-ID | 从 §2 中分配的 ID |
| 类型 | 集成（涉及 DB 或多层调用）/ 单测（纯逻辑） |
| 场景描述 | 具体的测试场景，包含输入和预期 |
| 覆盖规则ID | 对应的 BR-xx |
| 预期结果 | 期望的行为（成功/失败 + 具体表现） |

### §5.2 必测边界

从 PRD 规则中推导边界场景：
- 数量上限边界（如宠物数量 = 20 时再添加）
- 时间边界（如 token 刚好过期）
- 权限边界（非主人尝试主人操作）
- 并发场景（如同时加入同一家庭）
- 重复提交（如重复注册同一邮箱）

### §5.3 质量门禁

固定内容：
- checkstyle：通过
- 全量测试（含前置批次）：通过
- 核心验收用例：通过

## §6 代码组织

**数据来源**：CLAUDE.md（架构规范）+ 现有代码结构

1. 扫描 `specflow-api/src/main/java/com/specflow/api/modules/` 了解已有模块
2. 按 DDD Light 四层结构生成目录树
3. 如果涉及对已有代码的改造（如 P2 改造 P0 的 UserService），在「对已有代码的改造点」中列出
4. 如果存在跨模块依赖，在「跨模块依赖」中列出

## §7 快审记录

**生成时留空**，定稿时（模式 B）填写。

## §8 变更联动

**数据来源**：跨批次的依赖分析

生成规则：
1. 分析当前批次对前置批次的依赖（如 P2 需要修改 P0 的 users 表）
2. 分析当前批次对后续批次的预留（如 P0 预留 deleted 字段供 P2 使用）
3. 每个联动点分配 CL-xx 编号
4. 联动必须是双向的——如果 P2 Tech Pack 中有 CL-01（P2→P0），P0 Tech Pack §8 也必须记录对应条目
