# PRD-Lite 编写规范

## 总体原则

1. **纯业务视角** — 面向产品岗位，不涉及技术实现（数据库、API、框架等）
2. **决策导向** — 只记录"做什么"和"不做什么"，不写"怎么做"
3. **可追溯** — 每条验收标准可追溯到业务规则，每条规则有明确的优先级和批次

## 各章节编写要点

### §1 概述

- **文档范围**：用顿号分隔覆盖的功能领域
- **不覆盖**：明确排除项，避免范围蔓延
- 范例：
  > **文档范围**：用户注册登录、用户资料、宠物管理、家庭共享。
  > **不覆盖**：IoT 设备绑定、GPS/围栏/运动等业务功能、API 与数据模型设计。

### §2 角色与权限

- **角色定义表**：每个角色需要说明"获得方式"
- **角色关系**：如果一个用户可以同时拥有多种角色，必须在说明中明确
- **权限矩阵**：按操作对象分表，不按角色分表
  - 使用 ✅/❌ 标记
  - 每个操作对象单独一张表
  - 在表后添加补充说明
- **常见问题**：
  - 角色 vs 关系：区分"角色"（系统级身份）和"关系"（实体间的归属描述）
  - 范例：「"宠物主人"不是独立角色，而是用户与宠物之间的归属关系描述」

### §3 核心实体与关系

- **实体表**：每个实体一句话描述
- **关系图**：使用文本格式的 1-N、N-M 关系图
- **关系规则**：用列表说明归属、共享、可选性等规则
- **注意**：只写业务实体，不写技术表（如 session、migration 等）

### §4 功能需求

- **分组**：按功能领域分组（如"用户注册与登录"、"宠物管理"）
- **编号**：使用 `FR-4.x.x` 格式（4 是章节号，第一个 x 是领域序号，第二个 x 是功能点序号）
- **内容**：
  - 功能描述（用户能做什么）
  - 相关的业务规则引用
  - 字段要求（必填/选填、格式约束、默认值）
- **禁忌**：不写 API 路径、HTTP 方法、数据库字段名

### §5 业务规则

#### 5.1 关键约束

- **数量**：建议 10~20 条，只保留"必须正确实现"的规则
- **编号**：`BR-01` 起始，全文档唯一
- **四列**：编号、规则、优先级、批次
  - 优先级：规则本身的重要性（P0=核心、P1=重要、P2=增强）
  - 批次：该规则在哪个开发批次实现（可与优先级不同）
- **规则**：用陈述句，一条规则一个约束，避免复合规则

#### 5.2 边界场景

- 两列：场景、处理方式
- 处理方式要明确：拒绝/允许/提示内容
- 覆盖用户可能遇到的异常情况
- 尽量穷举，宁多勿少

### §6 验收标准

- **编号**：`AC-01` 起始，全文档唯一
- **按批次分组**：P0/P1/P2 各自一张表
- **四列**：AC-ID、验收描述、验收方式、覆盖规则
- **验收描述**：用"用户可以..."或"用户不能..."句式
- **验收方式**：自动（可自动化测试）/ 手工（需人工验证）
- **覆盖规则**：关联到 BR 编号，如无直接关联可填 `-`
- **关键检查**：每条 BR 至少被一条 AC 覆盖

### §7 交付约束与分期策略

#### 7.1 分期策略

- 表格四列：批次、范围、功能需求（FR 编号列表）、说明
- 如有注意事项（如批次间依赖），在表后补充
- 范例：
  > **P0 注意事项**：P0 批次暂不实现邮箱验证流程（P2），注册成功后用户直接可用。

#### 7.2 交付约束（给 Tech Pack）

- 按批次列出，每批次三项：
  1. 必须映射的规则：`BR-xx, BR-xx`
  2. 必须覆盖的验收项：`AC-xx ~ AC-xx`
  3. 必须覆盖的边界场景：列表形式
- 这一节是 PRD 和 Tech Pack 之间的合同，确保规则不丢失

### §8 未来扩展

- 列表形式，每项一句话描述
- 标注"本版本不实现"
- 帮助开发团队理解长期方向，避免短期设计挡路

## 编号规范汇总

| 类型 | 格式 | 示例 | 说明 |
|------|------|------|------|
| 功能需求 | FR-章节.领域.序号 | FR-4.1.1 | 章节号固定为 4 |
| 业务规则 | BR-序号 | BR-01 | 全文档唯一，两位数字 |
| 验收标准 | AC-序号 | AC-01 | 全文档唯一，两位数字 |
| 批次 | P0/P1/P2 | P0 | P0=核心，P1=扩展，P2=增强 |

## 写作风格

- 使用简洁的陈述句
- 涉及数值的规则要明确具体数字（如"最多 20 只"而非"有数量上限"）
- 枚举值要列出所有选项（如"种类：DOG、CAT"）
- 默认值要明确说明（如"不填则默认为邮箱 @ 前缀"）
- 时间限制要带单位（如"有效期 7 天"、"60 秒内"）
